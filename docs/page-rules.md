# ページ作成ルール

## 1ページ = 1概念

各ページは単一の概念・トピックを扱う。複数の概念をまとめない。

**OK：** `crs.md`（座標系とは何か）  
**NG：** `data.md`（ベクタ・ラスタ・フォーマットをまとめて説明）

## ファイル配置

```
src/content/docs/
├── index.mdx          # 全体の索引（トップページ）
├── <slug>.md          # トップレベルの概念
└── <category>/
    └── <slug>.md      # カテゴリ内の概念
```

- スラッグは英語・小文字・ハイフン区切り（例：`vector-data.md`）
- カテゴリは概念が3つ以上まとまるときに作る

## フロントマター（必須）

```yaml
---
title: （日本語タイトル）
description: （1〜2文の説明）
---
```

## 索引への追記

ページを追加したら `src/content/docs/index.mdx` の対応セクションにリンクを追加する。

```md
→ [ページタイトル](/gis/<slug>/)
```

## 「浅いページ」にしないための最低ライン

短い定義だけで終わるページは避ける。少なくとも、次の観点のうち4つ以上を含める。

- 定義
  対象が何で、何を表す概念なのかを冒頭で言い切る
- 必要性
  その概念がないと何が困るのか、どの場面で現れるのかを書く
- 構造
  要素、層、入れ子、データモデル、処理段階などの内訳を書く
- 入出力
  何を入力として受け、何を出力・保持するかを書く
- 制約
  仕様上の制約、表現できないこと、誤解されやすい点を書く
- 実装上の論点
  パース、変換、描画、保存、性能、互換性など、コードに関係する観点を書く
- 具体例
  実データ、XML/JSON断片、座標値、CLI、ライブラリAPIなどの具体例を書く
- 関連概念との差分
  似た概念との違い、選び分けの軸を書く
- 公式仕様への導線
  OGCやRFCなど、一次情報となる公式の仕様ページへのリンクを含める
- 仕様の具体的な説明
  用語の紹介だけでなく、要素、属性、構造、制約、バージョン差分などを具体的に説明する

## 推奨セクション

ページごとに見出し名は多少変えてよいが、次の流れを基本とする。

1. 概要
2. なぜ必要か
3. どういう構造か
4. どう使われるか
5. 実装で何を見るか
6. 制約・注意点
7. 具体例
8. 関連概念

## 技術的に深い記事にする観点

仕様やデータフォーマットのページでは、説明文だけで終わらせず、次の観点を優先する。

- どの要素が主役か
- どの識別子やキーで参照がつながるか
- どこが繰り返しになり、どこがネストになるか
- 省略可能な要素と必須要素は何か
- 実装時にどの順で読むと理解しやすいか
- 変換時に情報が落ちるポイントはどこか
- バージョン差分や互換性の注意点は何か
- データ量や性能に効く要因は何か
- 技術的な詳細を記述するときは、必要に応じて公式仕様の該当箇所へのリンクを貼る
  - 仕様全体のトップページだけでなく、説明している要素、構造、制約、バージョン差分に直接対応する節や章へのリンクを優先する
  - リンクはページ末の参考情報としてまとめるだけでなく、技術的な説明の中や、そのセクションの末尾にも置く
  - 読者が本文の説明と一次情報を対応付けられるように、リンク先の位置づけが分かる形で置く

## 例を入れるときのルール

- 例は短くてもよいが、実在しそうな形にする
- コードブロックを入れる場合は、何を見る例なのかを直前に書く
- XMLやJSONは、主役となる要素だけを抜き出してよい
- 例の後に「この例では何が重要か」を1〜3文で補足する

## 比較を書くときのルール

比較ページでなくても、読者が混同しやすい対象があるなら短い比較を入れる。

- 「何が違うか」だけでなく「その違いが実務で何に効くか」を書く
- 優劣ではなく、表現対象・用途・制約の違いとして書く
- 1つの表で比較できるなら、文章だけより表を優先する

## 実装者向けの視点

このドキュメントは、GIS初学者でもコードを読む・書く開発者を主な読者とする。したがって、概念説明だけでなく、次のどれかをできるだけ入れる。

- パーサでどのフィールドを見るか
- 変換処理でどこを保つべきか
- 表示ライブラリに渡すとき何へ落とし込むか
- デバッグ時にどの値を見るか
- サンプルデータを読むときの観察順

## 書き始める前のチェック

- このページの主語は1つに絞れているか
- 読者が「で、実際には何を見るのか」が分かる構成になっているか
- 定義の言い換えだけで段落を水増ししていないか
- 具体例、制約、実装観点のいずれかが不足していないか
- 関連ページに分割した方がよい話題を抱え込みすぎていないか
