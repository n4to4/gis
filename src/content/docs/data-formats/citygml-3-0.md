CityGML 3.0とは、3D都市モデルの意味論、ジオメトリ、関係、時間変化を概念モデルとして定義する標準。

## なぜ重要か

- 2.0までの「GML/XML交換形式中心」の理解から、実装技術に依存しにくい情報モデル中心へ整理された
- 建物外形だけでなく、内部空間、地物関係、時間変化、履歴管理まで標準の射程に入る
- 都市モデルをXML、データベース、JSON系表現へ写像するときの基準点になる
- 2.0データの移行や、PLATEAUのような既存資産を将来設計へつなぐ判断材料になる

## なぜ3.0が必要とされたのか

CityGML 2.0は広く使われた一方で、標準の中心がGMLエンコーディングに寄っていた。そのため、次のような論点が出やすかった。

- XML要素名と概念モデルが強く結びつき、別技術へ載せ替えるときに意味を保ちにくい
- LODが `lod1Solid` のような要素名で現れるため、詳細度と表現内容の関係を整理しにくい
- 屋内空間、占有空間、設備配置のような「空間そのもの」を地物と区別して扱いにくい
- 時間変化を持つ属性値と、都市モデル全体の版管理を標準だけで表しにくい
- BIMやシミュレーション系データとの接続点を、実装ごとに独自解釈しやすい

CityGML 3.0は、これらを踏まえて「まず概念モデルを標準化し、その上に複数のエンコーディングを載せる」という形へ寄せている。

## 何を表すのか

CityGML 3.0の主役は、個別のXML要素名ではなく、都市モデルを構成するクラス、属性、関連、空間表現の定義である。

| 観点 | 役割 | 実装時の見どころ |
| --- | --- | --- |
| 概念モデル | 都市地物をどう分類し、どう関連付けるかを定義する | XML要素ではなく、保持したい意味単位を先に決める |
| モジュール | Building、Transportation、Versioningのように責務を分ける | 必要なモジュールだけ実装対象にできる |
| 空間表現 | 屋外形状、屋内空間、占有空間などを表す | 地物と空間を別概念として扱う必要がある |
| 時間表現 | 地物の寿命、属性の時系列、都市モデルの版を表す | 変化の粒度が「時系列」か「版管理」かを分ける |
| エンコーディング | 概念モデルをGML/XMLなどへ写像する | 交換形式と内部モデルを分離して設計する |

## 2.0 と 3.0 の違い

| 観点 | 2.0 | 3.0 |
| --- | --- | --- |
| 標準化の中心 | GMLスキーマと交換形式 | UMLベースの概念モデル |
| 実装対象 | XML文書をどう読むかが中心 | 概念をどの技術へ写像するかが中心 |
| LOD | 0〜4で理解されることが多い | 0〜3に整理され、屋外と屋内の表現文脈が明確化 |
| 屋内表現 | LOD4に強く結びつく | Space概念で室・交通空間・占有空間を整理 |
| 動的データ | 標準だけでは弱い | Dynamizerで時系列値を関連付けやすい |
| 版管理 | 明示的な仕組みが弱い | Version / VersionTransitionで履歴を扱える |
| 技術展開 | XML前提が強い | GML/XML以外への展開を前提にしやすい |

この表のうち、LODの扱いは実装上の差が大きい。2.0では `lod2Solid` のように要素名へ詳細度が埋め込まれやすいが、3.0では「その地物が、どの文脈の空間表現を、どの詳細度で持つか」を概念上で整理しやすい。

## 主要モジュール

CityGML 3.0は、都市モデル全体を単一の巨大スキーマとしてではなく、責務ごとに分けたモジュール集合として理解すると追いやすい。

| モジュール | 役割 | 2.0から見た違い |
| --- | --- | --- |
| Core | 全モジュールの基礎。識別、関連、時間属性の基盤を持つ | 単なる共通要素より、概念的な土台としての位置付けが強い |
| Building | 建物、建物部位、境界面、開口部を表す | 屋内空間や構成関係との接続が整理しやすい |
| Construction | 橋梁やトンネルなどに共通する構築物の概念を扱う | 建築・土木系の共通性をまとめやすい |
| Transportation | 道路、鉄道、広場などを扱う | 交通空間の意味構造を他モジュールと揃えやすい |
| Versioning | 都市モデルの版、分岐、遷移を扱う | 2.0には薄かった履歴管理を標準化 |
| Dynamizer | 属性やジオメトリへ時系列値を関連付ける | センサーやシミュレーション結果との結び付きが明示的 |

## XMLとGMLでどう実装されるか

CityGML 3.0でエンコーディングが分離されたといっても、実務上はGML/XMLが中心になる場面が多い。OGCの標準も、Part 1が概念モデル、Part 2がGML Encoding Standardという二層構成になっている。

- 概念モデルは技術非依存のPIMとして定義される
- XML実装では、その概念モデルをGML 3.2ベースのスキーマへ写像する
- つまり「XMLが不要になった」のではなく、「XMLだけが標準の本体ではなくなった」と捉える方が近い
- 既存のツールチェーン、検証、交換運用では、引き続きXMLスキーマと名前空間の理解が重要になる

特に実装者の視点では、3.0でも次の点は引き続きXML前提で見ることになる。

| XML/GMLの観点 | 何を見るか | 実装時の見どころ |
| --- | --- | --- |
| 名前空間 | `core`、`bldg`、`vers` など | モジュール単位で責務を見分ける |
| ルート構造 | `core:CityModel` | 文書全体の入口を把握する |
| 識別子 | `gml:id`、`gml:identifier` | 版違いの地物と実体識別を分けて扱う |
| 参照 | `xlink:href` など | 幾何共有、履歴参照、関連地物の解決に使う |
| スキーマ | `core.xsd`、`building.xsd`、`versioning.xsd` など | どのモジュールをサポートするかを明確にできる |

仕様上、CityGML 3.0のGML EncodingはGML 3.2準拠で定義される。概念モデル上の `featureID` は、GML実装では `gml:id` として現れる。さらに、実体としての識別子を安定して持たせたい場合は `gml:identifier` を使う。この違いは2.0より重要で、版違いの同一地物を扱うときの基盤になる。

## LODと空間モデル

CityGML 3.0では、LODの数字だけでなく「何の空間を、どの粒度で表すか」が重要になる。

- 3.0のLODは、仕様上は4段階の LOD0〜LOD3 で整理される
- 2.0のLOD4に押し込まれがちだった屋内表現は、3.0ではSpace概念として独立して理解しやすい
- 建物そのもの、部位、境界面、部屋、交通空間、占有空間を別の意味単位として保持できる
- 同じ建物でも、外形表現と屋内空間表現を別の関心事として設計しやすい

実装では、「LOD番号をそのままUIの品質ラベルにする」のではなく、外観表示向けか、解析向けか、屋内経路向けかという用途に対応づけて使う方が安全である。

## 時間変化と版管理

3.0で技術的に大きいのは、変化を1種類として扱わない点である。

| 変化の種類 | 主な仕組み | 何を表すか |
| --- | --- | --- |
| 地物の存続期間 | Coreの時間属性 | その地物が現実世界やモデル上でいつ有効か |
| 属性値の時系列変化 | Dynamizer | 温度、消費電力、交通量、移動位置などの変化 |
| モデル全体の版や履歴 | Versioning | 更新前後の差分、分岐、履歴系列 |

一次情報のユーザーガイドでは、全地物に時間属性の基盤があり、Versioningが `Version` と `VersionTransition` を追加し、Dynamizerが地物の特定プロパティへ時系列値を結び付ける形で説明されている。2.0ではこの切り分けを標準だけで表すのが難しかった。

## XMLとして読むときの観察順

3.0をXML文書として読むなら、概念モデルを忘れずに次の順で追うと見通しがよい。

1. ルートの `core:CityModel` と名前空間宣言を見る
2. `core:cityObjectMember` 配下で主役の地物型を特定する
3. 地物の `gml:id` と `gml:identifier` を確認し、版識別と実体識別を分けて理解する
4. BuildingやTransportationモジュールの要素を追い、外形、部位、空間を読む
5. 履歴があればVersioningモジュール、時系列値があればDynamizerモジュールを追加で追う

2.0との違いは、`lod2Solid` のような「要素名を見れば意味が分かる」読み方だけでは足りない点にある。3.0では、同じXMLを読んでいても、背後の概念モデルを意識しないと、空間、版、時系列の責務を混同しやすい。

## 読み方

3.0のデータや仕様を読むときは、XMLタグ一覧から入るより、次の順で追うと理解しやすい。

1. まず対象がどのモジュールを使っているかを確認する
2. 次に、主役の地物が Building なのか Transportation なのかを確認する
3. その地物が持つ空間表現を、外形、部位、屋内空間のどれかに分けて追う
4. 動的値が必要なら Dynamizer、履歴が必要なら Versioning を読む
5. 最後に、その概念が実際のGML/XMLでどの要素へ写像されているかを確認する

## XML断片の例

建物をGML/XMLとして読むときの最小例は次のようになる。

```xml
<core:cityObjectMember>
  <bldg:Building gml:id="B1020_t1">
    <gml:identifier codeSpace="https://example.com/buildings">B1020</gml:identifier>
    <core:creationDate>2012-08-02T00:00:00</core:creationDate>
    <core:terminationDate>2013-10-09T00:00:00</core:terminationDate>
    <bldg:function>office</bldg:function>
  </bldg:Building>
</core:cityObjectMember>
```

この例では、`gml:id="B1020_t1"` が特定バージョンの地物を識別し、`gml:identifier` が実体としての建物識別子を表す。`core:creationDate` と `core:terminationDate` は、その版が有効な期間を示す。3.0のXMLを読むときは、タグ名だけでなく「これは版の識別か、実体の識別か」を切り分けて理解する必要がある。

## 実装で見るポイント

- 2.0互換のパーサーをそのまま延長するより、内部モデルをいったん概念ベースに置き直した方が移行しやすい
- `lod2Solid` のような要素名依存のモデルへ固定すると、3.0の空間概念を表しにくい
- 変換先がGeoJSONやglTFでも、地物型、境界面、空間階層、時間属性のどこを保持するかを先に決める必要がある
- Versioningまで扱う場合、単一レコード上書きではなく、履歴系列や分岐系列を保持できるデータベース設計が必要になる
- Dynamizerを扱う場合、静的属性と時系列属性を同じ列へ雑に押し込むと意味が失われやすい
- 3.0は概念モデルとエンコーディングが分かれているため、XML対応だけ実装しても仕様全体を理解したことにはならない
- XMLバリデーションだけ通っても、版管理や時系列値の意味が正しく保たれているとは限らない

## 具体例

建物の静的属性と動的属性を分けて扱うイメージは次のようになる。

```txt
Building
  id: bldg-1
  measuredHeight: 32.4
  spaces:
    - Room r-101
  dynamizers:
    - target: measuredHeight
      source: sensor/time-series
  versions:
    - v2024
    - v2025
```

この例で重要なのは、建物本体、屋内空間、時系列値、版管理が別の責務として並んでいる点である。2.0では1つのXML文書構造の中に押し込めて理解しがちだったが、3.0では内部モデルとして分けて考える方が自然である。

## 2.0から移行するときの論点

- 2.0のLOD4を、そのまま3.0へ機械的に写すと屋内空間の意味が曖昧になりやすい
- `cityObjectMember` と要素名中心で構築した読み取り処理は、3.0では抽象化不足になりやすい
- `gml:id` ベースの参照解決は引き続き重要だが、参照先の意味分類をより厳密に持つ必要がある
- 既存2.0資産を移行するときは、外形、意味属性、部位関係、時間情報のどれが欠落しているかを棚卸しする必要がある
- 3.0へ移す目的が「XML更新」なのか「都市モデルの運用改善」なのかで、必要な対応範囲が変わる

## 制約・注意点

- 3.0は2.0より整理されているが、そのぶん概念理解を飛ばしてXMLだけ追うと全体像を見失いやすい
- 既存ツールや公開データは2.0前提のものがまだ多く、実務では混在期間を前提にする必要がある
- 3.0対応をうたう実装でも、全モジュールを同じ深さでサポートするとは限らない
- 概念モデル中心の標準であるため、交換形式ごとの差異を無視すると互換性を誤認しやすい
