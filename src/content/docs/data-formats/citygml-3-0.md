CityGML 3.0とは、CityGML 2.0の課題を踏まえて再設計された都市モデル標準。

## なぜ重要か

- 2.0より柔軟なデータモデルで都市を表現できる
- 時間変化や版管理のような概念を扱いやすい
- 現実の都市業務に近い形で地物関係を整理しやすい
- 3D都市モデルを長期運用するための基盤になりやすい

## なぜ3.0が必要とされたのか

CityGML 2.0は広く使われた一方で、次のような課題があった。

- LODの扱いが分かりにくく、地物ごとの表現差を整理しにくい
- 地物間の関係や構成要素を柔軟に表すのが難しい場面がある
- 時間変化、バージョン、履歴のような情報を標準的に扱いにくい
- 実装や拡張の複雑さが高く、ユースケースごとの調整が増えやすい

CityGML 3.0は、こうした点を見直して、より拡張しやすく、運用しやすいモデルを目指した。

## 2.0 と 3.0 の違い

- 2.0はXMLスキーマ中心の印象が強いのに対し、3.0は概念モデルの整理が強化されている
- 2.0のLODは段階ごとの固定的な理解になりやすいのに対し、3.0は表現の意味づけが整理されている
- 2.0では時間変化の扱いが弱いのに対し、3.0ではダイナミックな変化を扱いやすい
- 2.0では地物のバージョンや履歴管理が標準として弱いのに対し、3.0では版管理の考え方が入っている
- 2.0よりも、3.0の方が地物間の関係や構成を明示しやすい

## 3.0の見どころ

CityGML 3.0では、交換形式そのものだけでなく、まず概念モデルを標準化する方向が明確になっている。これにより、GML/XMLだけに縛られず、別の実装技術へ展開しやすくなる。

| 観点 | 2.0 | 3.0 |
| --- | --- | --- |
| 標準化の中心 | GMLベースのエンコーディング | 概念モデルとエンコーディングの分離 |
| LOD | 0〜4の固定段階として理解されやすい | 表現対象との関係が整理される |
| 時間変化 | 弱い | DynamizerやVersioningで強化 |
| 室内空間 | 限定的 | Space概念により整理しやすい |
| 実装の自由度 | XML前提が強い | 複数技術へ展開しやすい |

## 3.0で追加・強化された機能

- Versioning
地物の版や更新履歴を扱うための考え方が強化された。

- Dynamizer
センサーデータや時系列値のような変化する属性を結びつけやすくなった。

- Spaceの導入
建物内部空間や占有空間のような概念を、より整理して表現しやすくなった。

- LOD概念の整理
LODの意味が見直され、地物ごとの表現をより明確に扱いやすくなった。

- モジュール構成の見直し
都市モデルの構成要素を、より拡張しやすい形で整理している。

## 実装で見るポイント

- 2.0のXML要素名を前提にした実装は、そのまま3.0へは移植しにくい
- 3.0では「どの概念を保持したいか」を先に整理してから、GML、JSON、DBスキーマなどの実装先を選ぶ方がよい
- 2.0から3.0へ変換する場合、LODの意味づけや地物関係の再整理が必要になる
- 既存データ資産が2.0中心なら、3.0の概念差分を把握したうえで段階的に移行する必要がある

## 具体例

3.0を読むときは、単一のXML文法としてだけでなく、「都市モデルの概念をどの単位で保持しているか」を見る必要がある。たとえば建物を扱う場合でも、外形、部位、内部空間、時間変化をそれぞれ独立した観点として設計しやすくなっている。

## 制約・注意点

- 3.0は改善点が多い一方で、既存ツールや既存データは2.0を前提にしている場合が多い
- 仕様理解なしに「3.0の方が新しいからそのまま置き換えられる」と考えると互換性の問題が出やすい
- 交換フォーマット、概念モデル、運用モデルを分けて理解しないと、仕様の利点が見えにくい
