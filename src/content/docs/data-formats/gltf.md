glTFとは、3Dモデルの形状、材質、テクスチャ、アニメーションを効率よく表現し、配布と描画に向けて最適化されたフォーマット。

## なぜ重要か

- WebGL、OpenGL、Vulkan、DirectX系の描画パイプラインへ載せやすい
- JSONとバイナリを組み合わせて、構造情報と数値配列を分けて保持できる
- テクスチャ、マテリアル、スキニング、アニメーションまで1つの形式で扱える
- Cesium、three.js、Babylon.js、Unity、Blenderなど、多くのツールとランタイムで対応が進んでいる
- 3D Tilesのタイル内容や、GIS系の3D可視化で中間形式として使われることが多い

## どこで利用されるか

glTFは、見た目を保ったまま3Dモデルを軽量に受け渡したい場面で使われる。

- ブラウザでの3D表示
  MapLibre系のカスタムレイヤー、CesiumJS、three.jsなどで読み込まれる
- ゲームエンジンやDCCツール間の受け渡し
  Blender、Unity、Unreal Engineなどの間でモデルを移すときに使われる
- GISと3D都市モデルの可視化
  CityGMLやIFCから変換したメッシュを表示向けに配布するときに使われる
- 3D Tilesのコンテンツ
  b3dmやi3dmでは、内部にglTFまたはGLBが入る
- AR/VR向け配信
  モデル、マテリアル、テクスチャをまとめて扱えるため、そのまま描画資産として流し込みやすい

## なぜ使われるか

glTFの利点は、単なる3Dファイル保存形式ではなく、描画に近い表現をそのまま渡せる点にある。

| 観点 | 利点 | 実装時の見どころ |
| --- | --- | --- |
| 配信効率 | JSON本体とバイナリ配列を分け、転送量を抑えやすい | 外部 `.bin` と埋め込み `data:` URL のどちらかを確認する |
| 読み込み速度 | 頂点配列やインデックスがGPU向けに近い形で並ぶ | `bufferView` と `accessor` の対応を追う |
| 表現力 | メッシュ、マテリアル、テクスチャ、アニメーションを一貫して持てる | どこまで実装対象にするかを先に決める |
| 相互運用性 | 多くの描画エンジンと変換ツールが対応する | 拡張仕様を使っているかを見る |
| 3D Tilesとの接続 | タイル配信とモデル表現を分担できる | 位置決めや属性は別レイヤーで持つ場合がある |

## 何を表すのか

glTFの主役は、地理座標そのものではなく、描画可能な3Dシーンである。

- ノード（Node）でシーン木を表す
- メッシュ（Mesh）で頂点群と面を表す
- マテリアル（Material）で見た目を表す
- テクスチャ（Texture）で画像資産を参照する
- スキン（Skin）とアニメーション（Animation）で変形や動きを表す

GIS文脈では、建物、樹木、設備のような地物をメッシュとして可視化するときに使われる。ただし、地物の意味論や空間参照系を標準で豊富に持つ形式ではない。

## JSONとバイナリの構造

glTFには、大きく2つの持ち方がある。

- `.gltf`
  JSONファイル本体と、外部 `.bin`、画像ファイルを分けて持つ形式
- `.glb`
  JSONチャンクとバイナリチャンクを1ファイルにまとめたバイナリ形式

どちらも論理構造はほぼ同じであり、違いは格納方法にある。

重要なのは、JSONとバイナリが同じ情報を二重に持っているのではない点である。

- JSONは構造と参照関係を持つ
- バイナリは大量の数値配列そのものを持つ

たとえば、頂点座標をJSON配列としてそのまま並べることもできるが、3Dモデルでは数万から数百万の数値になる。そのためglTFでは、数値本体はバイナリへ寄せ、JSON側には「どこに、どの型で、何個あるか」だけを書く。

この役割分担は、データベースでいえば「索引と実データ」、プログラムでいえば「メタデータとメモリ領域」の関係に近い。

- `meshes` が「このメッシュは `POSITION` と `NORMAL` を持つ」と宣言する
- `accessors` が「`POSITION` は `float32` の `VEC3` が 24 個ある」と定義する
- `bufferViews` が「その配列はバイト列のどの範囲か」を指す
- `buffers` が「実際のバイト列そのもの」を持つ

つまり、JSONを読むとデータ構造と参照先が分かり、バイナリを読むと実際の座標値やインデックス値が得られる。

| 要素 | 役割 | 実装時の見どころ |
| --- | --- | --- |
| `scenes` | シーン全体の入口 | 既定シーンがどれかを確認する |
| `nodes` | 変換行列を持つシーン木 | 親子関係と `translation` / `rotation` / `scale` を追う |
| `meshes` | 描画対象のメッシュ集合 | `primitives` ごとにマテリアルや属性が分かれる |
| `buffers` | 生のバイト列 | 外部ファイルかGLB内部かを確認する |
| `bufferViews` | バイト列の切り出し範囲 | オフセット、長さ、用途を確認する |
| `accessors` | 数値配列としての解釈 | 型、個数、最小値・最大値を確認する |
| `materials` | PBR中心の材質定義 | ベースカラー、金属度、粗さ、テクスチャ参照を見る |
| `images` / `textures` | 画像資産と参照関係 | MIME型、URI、サンプラー設定を確認する |

## バイナリはどう入っているか

頂点座標や法線、UV、インデックスは、JSONに直接並べるのではなく、`buffers` に生バイト列として保存される。その切り出し単位が `bufferViews`、数値配列としての意味付けが `accessors` である。

1. `buffers` がバイト列全体を持つ
2. `bufferViews` がその一部範囲を切り出す
3. `accessors` が要素型と個数を与える
4. `meshes[].primitives[].attributes` が `POSITION` や `NORMAL` として参照する

この3段階を理解しないと、glTFのバイナリ構造は追いにくい。

`.gltf` と `.glb` では、このつながり方の見え方だけが少し違う。

- `.gltf`
  JSON本体の中で `buffers[].uri` が外部の `.bin` ファイルや埋め込み `data:` URL を指す
- `.glb`
  JSONチャンクの中では `buffer` の定義を持ちつつ、実際のバイト列は同一ファイル内の BIN チャンクに続く

どちらの場合でも、アプリケーションがやることは同じである。先にJSONを読み、次に `buffer` が指す生バイト列を見つけ、最後に `bufferView` と `accessor` で意味付けする。

たとえば、頂点座標は次のようにつながる。

```json
{
  "meshes": [
    {
      "primitives": [
        {
          "attributes": {
            "POSITION": 0
          },
          "indices": 1
        }
      ]
    }
  ],
  "accessors": [
    {
      "bufferView": 0,
      "componentType": 5126,
      "count": 24,
      "type": "VEC3"
    }
  ],
  "bufferViews": [
    {
      "buffer": 0,
      "byteOffset": 0,
      "byteLength": 288
    }
  ],
  "buffers": [
    {
      "byteLength": 288
    }
  ]
}
```

この例では、`POSITION: 0` が `accessors[0]` を参照し、その `accessor` が `bufferViews[0]` を経由してバイト列の先頭 288 バイトを `float32` の `VEC3` 配列として解釈する。

`.gltf` であれば、この 288 バイトは外部 `.bin` ファイルの先頭部分にあるかもしれない。`.glb` であれば、同じ 288 バイトが同一ファイル内の BIN チャンクに置かれているだけである。論理的な参照のしかたは変わらない。

## GLBはどういうバイナリフォーマットか

`.glb` は、glTFの内容を1ファイルへまとめたコンテナ形式である。

- 先頭にヘッダーを持つ
- その後ろに JSON チャンクが続く
- 必要に応じて BIN チャンクが続く

概念的には次のような並びになる。

```txt
[header][JSON chunk][BIN chunk]
```

ヘッダーでは、ファイル識別子、バージョン、全体長を持つ。JSONチャンクには通常のglTF JSONが入り、BINチャンクには頂点・インデックス・その他バイナリ配列が入る。

実装上は、まずGLBヘッダーを読み、JSONチャンクをパースし、その後 `buffer` が指すバイナリ領域を `bufferView` と `accessor` で解釈する流れになる。

## 地物や属性はどう保存されるか

glTFは、CityGMLやGeoJSONのように「地物（Feature）中心」の標準ではない。基本単位は、描画対象としてのノード、メッシュ、プリミティブである。

そのため、GISの地物や属性は次のような形で保存されることが多い。

| 保存先 | 何を持てるか | 制約 |
| --- | --- | --- |
| `nodes` | オブジェクト単位の名前、変換、階層 | 地物型や属性スキーマは標準化されない |
| `meshes` | 形状そのもの | 同じメッシュを複数ノードで再利用できる |
| `materials` | 見た目の属性 | GIS属性ではなく描画属性が中心 |
| `extras` | 任意のJSON属性 | ツール間で意味が保証されない |
| 拡張仕様 | 独自または標準拡張の属性 | 対応実装が限定される場合がある |

単純な例では、建物IDや名称のような属性を `node.extras` へ入れることがある。

```json
{
  "nodes": [
    {
      "name": "building-42",
      "mesh": 0,
      "extras": {
        "featureId": "bldg-42",
        "measuredHeight": 32.4,
        "usage": "office"
      }
    }
  ]
}
```

この例では、`mesh` が形状、`extras` が属性の入れ物である。ただし、`extras` は「何をどう入れるか」が標準で厳密には決まっていないため、解析側で個別対応が必要になる。

## GISで使うときの論点

- glTF自体はCRSを標準で強く規定しないため、座標の意味を外部仕様や上位コンテナに委ねることが多い
- 緯度経度や投影座標をそのまま巨大な値で持つと、描画精度やローカル原点の扱いが問題になりやすい
- 地物属性を豊富に保持したい用途では、glTF単体では意味論が不足しやすい
- 逆に、表示向けに形状と見た目を高速に配りたい用途では相性がよい
- CityGMLやIFCから変換するときは、意味論、部位階層、属性のどこを捨てるかが論点になる

## 読み方

glTFデータを読むときは、次の順で追うと構造を見失いにくい。

1. まず `scene` と `scenes` を見て、どのシーンが入口か確認する
2. `nodes` を追い、どのノードがどの `mesh` を参照するか確認する
3. `meshes[].primitives` を見て、どの属性があるか確認する
4. `accessors`、`bufferViews`、`buffers` をたどって、実バイト列の位置と型を確認する
5. 属性を見たい場合は `extras` や拡張仕様の有無を確認する

## 制約・注意点

- glTFは表示向けの形式であり、GISの意味論やトポロジを豊富に持つ形式ではない
- 属性を `extras` に入れても、他ツールで保持される保証は弱い
- 1つの大きな都市モデルをそのまま単一glTFへ入れると、読み込みや更新単位が粗くなりやすい
- 精密な測地座標を直接持つより、ローカル座標へ落として上位で位置決めした方が扱いやすい場合が多い
- 実データでは `KHR_draco_mesh_compression` や `EXT_meshopt_compression` などの拡張が使われ、素朴なローダーでは読めないことがある

## 関連概念との差分

| 形式 | 主役 | 向いている用途 | glTFとの違い |
| --- | --- | --- | --- |
| CityGML | 地物と意味論 | 都市モデルの交換、分析、保持 | glTFより意味情報が豊富で、表示には重い |
| 3D Tiles | タイル配信 | 広域3Dデータの段階配信 | glTFはタイル分割や空間索引を持たない |
| MVT | 2Dベクタタイル | 地図表示の高速化 | glTFは3Dメッシュ中心で、2D地図の地物表現とは異なる |
| OBJ | 単純なメッシュ | 形状の受け渡し | glTFは材質、テクスチャ、シーン構造を持ちやすい |
