3D Tilesとは、巨大な3D地理空間データをタイル単位に分割し、階層化して段階的に配信・描画するための形式。

バージョンごとの差分は、[3D Tiles のバージョン差分](/gis/data-formats/3d-tiles-versions/) に分けて整理する。

## なぜ重要か

- フォトグラメトリ、3D建物、点群、BIM/CAD由来データのような重い3Dデータを一括読み込みせずに扱える
- 視点や距離に応じて必要なタイルだけを読むため、描画性能と通信量を両立しやすい
- 1つの巨大モデルではなく、空間分割された集合として扱うため、都市規模の可視化に向く
- 地物ごとのメタデータを持たせることで、選択、検索、スタイリング、属性参照につなげやすい
- Cesium系の可視化だけでなく、3D地理空間データの公開・配信形式として参照されることが多い

## 何を表すのか

3D Tilesの主役は、単一の3Dモデルではなく、タイル集合全体を記述する `tileset` である。

- `tileset.json` がタイル木の入口になる
- 各タイルは空間範囲、詳細度の切り替え条件、子タイル、コンテンツ参照を持つ
- タイルの中身は描画可能なデータで、3D Tiles 1.1では glTF が主な形式になる
- 点群、インスタンス配置、フォトグラメトリのように性質の異なる3Dデータも、同じ階層構造の中へ載せられる

つまり3D Tilesは、形状そのものを定義するフォーマットであると同時に、「どの範囲のデータを、どの詳細度で、どう分割して配るか」を定義する配信形式でもある。

## 基本構造

3D Tilesを読むときは、`tileset`、`tile`、`content`、`boundingVolume`、`geometricError` の役割を分けると追いやすい。

| 項目 | 役割 | 実装時の見どころ |
| --- | --- | --- |
| `asset` | 仕様バージョンなどのメタ情報 | `version` が `1.0` か `1.1` かを確認する |
| `root` | タイル木の根 | 全体の入口として最初に読む |
| `boundingVolume` | タイルが覆う空間範囲 | ボックス、領域、球のどれで表すかを確認する |
| `geometricError` | 子へ降りるべきかを決める誤差指標 | 距離ベースLOD切り替えの基準になる |
| `refine` | 親子タイルの切り替え方法 | 置換か加算かで描画挙動が変わる |
| `content.uri` | 実際のタイル内容への参照 | glTF、GLB、または旧形式を指す |
| `children` | 下位タイル | 空間分割の木構造を作る |
| `metadata` | タイルや地物に付く属性 | スタイリングや検索に使う |

## タイルの中身は何か

3D Tilesでは、タイル木とタイル内容を分けて考える必要がある。

- タイル木は `tileset.json` が持つ
- タイル内容は各 `content.uri` が指す
- 3D Tiles 1.1では glTF 2.0 が主なタイル内容である
- `b3dm`、`i3dm`、`pnts`、`cmpt` は 1.0 で使われた旧形式で、1.1では非推奨扱いになっている

この分離が重要なのは、3D Tilesが「1つのファイル形式」ではなく、「空間木と中身の組み合わせ」で成り立つからである。glTFは1タイルの描画内容を表し、3D Tilesはそれを大量配信するための外側の構造を担当する。

## 読み方

実データを読むときは、次の順で追うと理解しやすい。

1. まず `asset.version` を見て、1.0系か1.1系かを確認する
2. `root.boundingVolume` と `root.geometricError` を見て、全体の空間範囲とLODの基準を把握する
3. `children` を辿って、空間木がどの粒度で分割されているかを見る
4. 各タイルの `content.uri` を見て、中身が glTF / GLB か旧形式かを確認する
5. 属性参照が必要なら、メタデータやスタイル定義の有無を見る

## 実装で見るポイント

- 単に `content.uri` を順に読むだけでは足りず、`geometricError` と視点距離に応じた読み込み制御が必要になる
- タイル内容が glTF でも、ワールド座標への配置はタイル木側の情報と合わせて解釈する必要がある
- 旧形式の `b3dm` や `i3dm` を読む実装では、1.1の glTF中心構成への移行を見据えた分離が必要になる
- 暗黙タイル分割（implicit tiling）を使うデータでは、子タイルがJSONへ明示列挙されないため、テンプレートURIと部分木の理解が必要になる
- 地物選択や色分けを実装するなら、形状だけでなくメタデータの保持方法まで決める必要がある
- CityGMLやIFCから変換した場合、意味情報の一部はタイル内容ではなく外側のメタデータへ移ることがある

## 具体例

最小の `tileset.json` は次のようになる。

```json
{
  "asset": { "version": "1.1" },
  "geometricError": 500,
  "root": {
    "boundingVolume": {
      "box": [0, 0, 0, 100, 0, 0, 0, 100, 0, 0, 0, 60]
    },
    "geometricError": 100,
    "refine": "REPLACE",
    "content": { "uri": "tiles/root.glb" },
    "children": [
      {
        "boundingVolume": {
          "box": [50, 0, 0, 50, 0, 0, 0, 100, 0, 0, 0, 60]
        },
        "geometricError": 0,
        "content": { "uri": "tiles/east.glb" }
      }
    ]
  }
}
```

この例では、`root` が全体タイル、`children` が下位タイル、`content.uri` が描画内容を指す。重要なのは、メッシュ本体が `root.glb` や `east.glb` にあり、LODの切り替え条件や空間範囲は `tileset.json` 側が持つ点である。

## バージョン差分

3D Tiles では、1.0 から 1.1 で glTF 中心の構成へ重心が移っている。さらに、Cesium の 2026-03-03 の OGC Member Meeting 資料では、glTF 基盤を強めた 2.0 方向も示されている。

このページでは形式そのものの骨格を主に扱い、版ごとの差分は [3D Tiles のバージョン差分](/gis/data-formats/3d-tiles-versions/) にまとめる。

| 観点 | 1.0 | 1.1 |
| --- | --- | --- |
| 主なタイル内容 | `b3dm` `i3dm` `pnts` `cmpt` | glTF中心 |
| メタデータ | 制約が大きい | 構造化メタデータを扱いやすい |
| 複数コンテンツ | 前提が弱い | タイルが複数コンテンツを持てる |
| 暗黙タイル分割 | なし | あり |
| 将来性 | 互換維持の対象 | 現行仕様の中心 |

既存データでは 1.0 系の形式にまだ遭遇するが、新規実装では 1.1 の glTF 中心構成を前提にした方が整理しやすい。

## 制約・注意点

- 3D Tilesは配信と描画最適化の形式であり、CityGMLのような意味論中心の交換形式とは役割が違う
- 1つのタイルに何を詰めるか、どう分割するかで性能が大きく変わるため、変換時のタイル設計が重要になる
- CRSや座標変換を曖昧にしたまま生成すると、表示位置のずれや精度問題につながる
- `tileset.json` を読めても、glTF、メタデータ、スタイル、暗黙タイル分割まで対応しないと実用上は不十分になりやすい
- 旧形式対応が必要なプロジェクトでは、1.0と1.1を同じ抽象化で扱えるかが設計上の分かれ目になる
